# 工单管理系统实施计划 (Implementation Plan)

该项目的实施分为 6 个主要阶段，从系统底层搭建到全链路 AI 自动化直至交付。

## 阶段 1：项目初始化与基础设施 (Phase 1: Foundation Setup)
- [x] Initialize monorepo structure (`/client`, `/server`)
- [x] Set up Express server with TypeScript
- [x] Set up React app with TypeScript
- [x] Set up PostgreSQL database (Prisma initialized)

## 阶段 2：邮件收发核心与基础工单 (Phase 2: Email & Ticket Basics)
- [ ] **2.1 SMTP 发件模块开发:** 通过 `Nodemailer` 配置外部企业邮箱，封装好发件核心方法。
- [ ] **2.2 IMAP 邮件监听脚本:** 使用 `imapflow` 搭建长时间监听程序或轮询脚本，把外部客户邮件同步拉取为服务端数据。
- [ ] **2.3 核心工单转换 API:** 将获得的新邮件内容正规化转换为数据库里的 `Ticket` 与对话记录。
- [ ] **2.4 审计日志基础拦截:** 编写一套底层追踪函数，监控工单的任何属性变更并写入日志（供后期追溯时间线）。

## 阶段 3：直属知识库与向量化引擎 (Phase 3: Knowledge Base & VectorDB)
- [ ] **3.1 知识库后台 CRUD:** 实现知识库（QA 文档/业务策略）后台管理的增、查、改、删核心 API。
- [ ] **3.2 `pgvector` 存储支持:** 初始化 PostgreSQL 的向量扩展配置。
- [ ] **3.3 内容 Embeddings 嵌入:** 将存入的新知识库内容，通过 OpenAI API 数字化转换为向量信息并落库，使其具备之后供给 AI 搜索的能力。

## 阶段 4：AI 自动化决策引擎 (Phase 4: AI Workflow & Automation)
- [ ] **4.1 自动打标与分类功能:** 新邮件入库后，发送文本到 AI 解析并强制归类（General/Technical/Refund）及生成前情摘要。
- [ ] **4.2 向量语境召回:** 将新邮件内容进行向量查询，从数据库调取出对应的“知识库参考”。
- [ ] **4.3 分叉判定引擎:**
      - **A)** 知识库支持充足：AI 生成最终回复并直接自动触发 SMTP 外发逻辑，关单或更新状态。
      - **B)** 超纲问题或兜底：判断未能解决时，平均接手分配给人工代理，原 AI 生产的内容存入该工单的“建议回复库”。
- [ ] **4.4 SLA 报警定时器:** 设计一个监控定时任务引擎（BullMQ等），追踪距离未回复时间 > 24H的工单，自动修改特定标识列引发警报。

## 阶段 5：Dashboard 及交互视图 (Phase 5: User Interface)
- [ ] **5.1 框架结构设计:** 开发带有左侧菜单栏和顶部快速操作区的后台标准布局。
- [ ] **5.2 核心工单看板与详情:** 开发具备排序过滤能力的工单列表页；在详情页完整渲染邮件聊天历史、状态流转条、超时标红并展示前置生成的 AI 建议回复草稿。
- [ ] **5.3 审计与权限系统 UI:** 直观展示出“某工单时间线上的历代动作(Audit Logs)”，并支持多开下级代理账号的页面。
- [ ] **5.4 性能数据面板 (Metrics Dashboard):** 读取处理历史的聚合数据，开发出图表用以汇总工单解决率与 AI 自动化覆盖率。

## 阶段 6：测试及部署试运行 (Phase 6: QA & Deployment)
- [ ] **6.1 全功能端到端模拟联调:** 利用测试企业邮箱收发，模拟完整客户流测试工单流转系统。
- [ ] **6.2 上线前部署迁移:** 部署应用节点、数据库连接调整并将自动化脚本部署至常驻进程/云环境。
